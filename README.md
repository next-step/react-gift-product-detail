### 테스트별 시나리오

#### 로그인 페이지 테스트 시나리오

- 시나리오 1: 로그인 폼 초기 렌더링
  - 목적: 로그인 페이지의 기본 UI 요소가 정상적으로 렌더링되는지 확인
  - 조건: 페이지 초기 진입
  - 예상 결과:
    - 이메일 입력 필드가 화면에 표시된다.
    - 비밀번호 입력 필드가 화면에 표시된다.
    - 로그인 버튼이 화면에 표시되어 비활성화 상태이다.

- 시나리오 2: 유효하지 않은 입력 상태
  - 목적: 입력값이 유효하지 않을 경우 로그인 버튼이 비활성화되는지 확인
  - 조건: 이메일 또는 비밀번호가 빈 문자열 또는 유효하지 않은 값
  - 예상 결과: 로그인 버튼은 disabled 상태이다.

- 시나리오 3: 유효한 이메일/비밀번호 입력 시 버튼 활성화
  - 목적: 입력값이 유효할 경우 로그인 버튼이 활성화되는지 확인
  - 조건:
    - 이메일 입력: test@kakao.com
    - 비밀번호 입력: 12345678
  - 예상 결과:
    - 이메일과 비밀번호 유효성 검사를 통과한다.
    - 로그인 버튼이 활성화된다.

- 시나리오 4: 로그인 성공 시 유저 상태 업데이트
  - 목적: 로그인에 성공하면 유저 정보가 setUser로 전달되는지 확인
  - 조건:
    - 이메일: tester@kakao.com
    - 비밀번호: 12345678
  - 예상 결과:
    - setUser가 호출된다.
    - 인자로 전달된 객체에 이메일, 이름, 토큰이 포함된다.

- 시나리오 5: 유효하지 않은 입력 상태
  - 목적: 로그인에 실패했을 때 사용자에게 에러 메시지를 보여주는지 확인
  - 조건:
    - 이메일: fail@fail.com
    - 비밀번호: 12345678
  - 예상 결과:
    - 토스트로 "@kakao.com 이메일 주소만 가능합니다." 메시지가 표시된다.

#### 선물 랭킹 섹션 테스트 시나리오

- 시나리오 1: 초기 렌더링 시 제목과 로딩 스피너 표시
  - 목적: 기본 UI 요소와 초기 로딩 상태 확인
  - 조건: URL에 targetType=ALL, rankType=MANY_WISH가 포함된 상태로 페이지 진입
  - 예상 결과:
    - "실시간 급상승 선물랭킹" 제목이 표시된다.
    - 로딩 스피너가 나타난다.
    - 상품 데이터가 로드된 후 이미지("부드러운 고구마 라떼 케이크")가 표시된다.

- 시나리오 2: 필터 및 탭 변경 시 변경된 상품 표시
  - 목적: 필터나 탭을 클릭하여 상품 목록이 업데이트되는지 확인
  - 조건: URL에 targetType=FEMALE, rankType=MANY_GIFT로 설정 후 렌더링
  - 예상 결과:
    - "황올반+BBQ양념반+콜라1.25L" 상품 이미지가 표시된다.

- 시나리오 3: 상품이 없을 때 안내 메시지 표시
  - 목적: 필터 조합 결과가 없을 경우 적절한 안내 메시지를 표시하는지 확인
  - 조건: targetType=TEEN, rankType=MANY_WISH로 설정된 상태이며 서버 응답 데이터는 빈 배열
  - 예상 결과: "상품이 없습니다."라는 메시지가 표시된다.

- 시나리오 4: 더보기 버튼 클릭 시 상품이 추가로 렌더링됨
  - 목적: 기본 6개만 표시된 상품 목록에서 "더보기" 클릭 시 전체 상품이 보이는지 확인
  - 조건: targetType=ALL, rankType=MANY_WISH로 설정 후 렌더링
  - 예상 결과:
    - 초기에 상품 이미지가 6개로 렌더링된다.
    - "더보기" 버튼 클릭 시, 이미지가 추가로 렌더링되어 모든 상품을 보여준다.

#### Input 컴포넌트 테스트 시나리오

- 시나리오 1: 기본 렌더링
  - 목적: placeholder와 입력 필드가 정상적으로 렌더링되는지 확인
  - 조건: error 없이 placeholder만 전달된 상태
  - 예상 결과: placeholder 텍스트가 있는 input 필드가 화면에 표시된다.

- 시나리오 2: 입력 이벤트 처리
  - 목적: 사용자가 입력한 값을 onChange 핸들러가 정상적으로 처리하는지 확인
  - 조건: onChange mock 함수와 함께 렌더링
  - 예상 결과: 키 입력 시 onChange 함수가 호출된다.

- 시나리오 3: 에러 메시지 표시
  - 목적: error prop이 있을 경우 에러 메시지가 렌더링되는지 확인
  - 조건: error prop으로 문자열 전달
  - 예상 결과: error 메시지가 화면에 표시된다.

- 시나리오 4: onBlur 동작 확인
  - 목적: 포커스 해제 시 onBlur 이벤트가 발생하는지 확인
  - 조건: onBlur mock 함수를 전달하고 input 필드를 클릭 후 다른 요소로 포커스 이동
  - 예상 결과: onBlur 함수가 호출된다.

  #### Modal 컴포넌트 테스트 시나리오

- 시나리오 1: 모달 내용 렌더링
  - 목적: children으로 전달된 콘텐츠가 화면에 정상적으로 표시되는지 확인
  - 조건: 텍스트를 포함한 children 전달
  - 예상 결과: 전달된 텍스트 또는 콘텐츠가 화면에 렌더링된다.

- 시나리오 2: 오버레이 클릭 시 닫힘 처리
  - 목적: 모달 바깥 영역(오버레이)을 클릭했을 때 onClose 핸들러가 호출되는지 확인
  - 조건: onClose mock 함수 전달, ModalOverlay 클릭
  - 예상 결과: onClose 함수가 호출된다.

- 시나리오 3: 콘텐츠 클릭 시 닫힘 방지
  - 목적: 모달 안쪽 콘텐츠 영역 클릭 시 이벤트 전파가 중단되어 onClose가 호출되지 않는지 확인
  - 조건: onClose mock 함수 전달, ModalContent 클릭
  - 예상 결과: onClose 함수가 호출되지 않는다.
